================================================================================
A) SYSTEM OVERVIEW
================================================================================

O que é:
  SIGA (Sistema Integrado de Gestão de Ativos) é uma aplicação web industrial
  mobile-first construída para a empresa "Pro Raça". É inteiramente client-side
  (sem backend próprio) e usa Firebase como único serviço de nuvem.

Propósito principal:
  Digitalizar e centralizar cinco áreas operacionais de uma fábrica:
    1. Rastreamento de manutenção de maquinário industrial
    2. Inspeção de veículos da frota (caminhões, carros, motos)
    3. Auditorias de limpeza de facilities (metodologia 5S)
    4. Ordens de serviço, pedidos de compra e geração de documentos
    5. Gestão de fornecedores, autenticação por papel e dashboard de KPIs

Tipo de sistema:
  Plataforma de Operações Industriais / Sistema de Coleta de Dados em Campo +
  Dashboard de Gestão com controlo de acesso por papel (admin, maintenance,
  operations, purchasing). Evoluiu de MVP para plataforma de produção estruturada.

Estado de evolução:
  v1.0: Módulos operacionais (maquinário, frota, limpeza) com autenticação simulada.
  v2.0: Módulos de OS e Compras separados, geração de documentos, lista dinâmica
        de materiais, sistema unificado work_orders.
  v2.1: Firebase Auth real, RBAC, dashboard de KPIs, fornecedores,
        workflow de aprovação de compras, regras de segurança Firestore,
        bridge de migração de dados, Cloud Functions readiness completo.
  v2.2 (atual): Estabilização de todos os módulos operacionais.
        Correção de 5 race conditions DOMContentLoaded, checkAuth em app-painel-frota.js,
        remoção do window.onclick global, event delegation em todos os modais e
        formulários dinâmicos, substituição de alert() por toast flutuante.


================================================================================
B) ARCHITECTURE
================================================================================

Padrão Geral de Arquitetura:

  Static HTML Pages (UI)
          ↓
  ES Module JS — app-*.js (Business Logic / Page Controllers)
          ↓
  Core DB Layer — db-*.js (Firebase abstraction + domain logic)
          ↓
  Firebase Auth | Firestore (8 coleções) | Storage (fotos)

Decisões arquiteturais chave:

  - SEM build system. HTML puro + ES Modules carregados diretamente pelo
    navegador via <script type="module">. Zero bundlers, zero transpiladores.

  - Firebase SDK via CDN (gstatic.com/firebasejs/10.7.1). Sem npm em runtime.

  - firebase-config.js exporta db, storage e auth. Todos os módulos importam daqui.

  - Camada DB modular e separada por domínio:
      js/core/db.js             — módulo legado (maquinário, frota, limpeza)
      js/core/db-os.js          — work orders (manutenção + serviço), docs SRV
      js/core/db-compras.js     — pedidos de compra, docs PO, query unificado,
                                  workflow de aprovação completo
      js/core/db-unified.js     — sistema unificado de work_orders v2: KPIs,
                                  automações, numeração por timestamp
      js/core/db-auth.js        — Firebase Auth + RBAC (4 papéis), checkAuth()
      js/core/db-bridge.js      — bridge para leitura dupla (legacy + work_orders)
                                  + utilitário de migração
      js/core/db-dashboard.js   — agregações de KPIs para o painel de gestão
      js/core/db-fornecedores.js — CRUD de fornecedores + estatísticas

  - Dados mestres estáticos (js/data/) — máquinas, veículos, zonas de limpeza
    armazenados como arrays JS. Atuam como catálogo local (substituto de ERP).

  - Firebase Auth implementado com papéis lidos do Firestore (users/{uid}.role).
    Sem custom claims — papel na subcoleção users para simplicidade client-side.

  - Numeração de documentos por timestamp: SRV-YYYYMMDD-XXXX (db-unified.js)
    elimina risco de condição de corrida dos contadores sequenciais anteriores.
    (Nota: db-compras.js ainda usa contador sequencial — refatoração pendente.)

Estrutura de pastas:
  index.html                  → Hub de módulos + barra de sessão
  auth/                       → Página de login
  dashboard/                  → Painel de gestão
  maquinario/                 → Módulo de maquinário (4 páginas)
  frota/                      → Módulo de frota (3 páginas)
  limpeza/                    → Módulo de limpeza (3 páginas)
  os/                         → Ordens de Serviço + Documentos (3 páginas)
  compras/                    → Pedidos de Compra (2 páginas)
  js/core/                    → Camada de acesso ao Firebase (8 ficheiros)
  js/data/                    → Catálogos mestres estáticos
  js/modules/                 → Um app-*.js por página
  css/                        → global.css + stylesheets por módulo
  firestore.rules             → Regras de segurança por papel

D) DATA FLOW
================================================================================

Firestore Collections Map (v2.1):
  historico_manutencao  → Registros de OS de maquinário (formato legado)
  auditorias_limpeza    → Registros de auditoria 5S com checklist + fotos
  checklists_frota      → Registros de inspeção de veículos
  work_orders           → Sistema unificado de OS (manutenção|serviço|inspeção|limpeza)
  purchase_orders       → Pedidos de compra com workflow de aprovação
  documents             → Documentos gerados (SRV-XXXX | PO-XXXX)
  users                 → Perfis de utilizadores + papéis (RBAC)
  fornecedores          → Base de dados de fornecedores
  (kpi_cache)           → Cache de KPIs pré-calculados [reservado para CF Fase 2]

Fluxo de dados genérico (todos os módulos):
  1. checkAuth(paginaId) — guard de autenticação e papel; redirect se não autorizado
  2. Página abre, URL params analisados
  3. Dados mestres estáticos carregados de js/data/ (sem chamada de rede)
  4. Leitura Firebase: getDocs / getDoc na coleção relevante
  5. Registros ordenados client-side por timestampEnvio (DESC)
  6. HTML renderizado dinamicamente via innerHTML / createElement
  7. Interação do utilizador (submit, clique)
  8. Validação client-side → se falhar: toast ou alert
  9. Escrita Firebase: addDoc / updateDoc
  10. Upload de foto para Storage (se aplicável) → URL no Firestore
  11. Sucesso: redirect ou toast + atualização do DOM

Estratégia de Timestamp (dupla — mantida de v2.0):
  - serverTimestamp() (Firestore) — para auditoria e confiabilidade
  - Date.now() como timestampEnvio — para ordenação client-side eficiente
  Necessário porque serverTimestamp() não é comparável antes da resolução async.

E) KEY FUNCTIONALITIES
================================================================================

--- Autenticação e RBAC (NOVO v2.1) ---

  checkAuth(paginaId):
    Chamado no topo de cada página protegida. Se não autenticado, redireciona
    para auth/login.html calculando o path relativo automaticamente.
    Se papel não tem permissão para a página, redireciona para index.html com alert.

  Perfil de utilizador:
    { uid, email, nome, role, ativo } — lido de users/{uid} no Firestore.
    Cacheado em memória (_currentProfile) para evitar leituras redundantes.

  Login flow:
    signInWithEmailAndPassword → _carregarPerfil() → verificar ativo → updateDoc lastLogin
    Se conta desativada: signOut() automático + erro.

  Redirect inteligente:
    _resolverCaminhoLogin() calcula "../" prefix baseado na profundidade da URL
    para funcionar de qualquer subpasta (os/, compras/, dashboard/, etc.)

--- Workflow de Aprovação de Compras (NOVO v2.1) ---

  Timeline auditada:
    Cada ação (aprovar, rejeitar, enviar, receber, comentar) adiciona entrada
    ao array timeline do purchase_order com { acao, usuario, uid, timestamp, nota }.

  Guardrails por status:
    aprovarPedido — apenas se status === "pending"
    rejeitarPedido — motivo obrigatório (mín. 5 chars), não aceita received/cancelled
    marcarPedidoEnviado — apenas se status === "approved"
    marcarPedidoRecebido — apenas se status === "ordered"

  Regras Firestore complementares:
    Apenas roles purchasing | admin podem mudar campos de status e aprovadoPor.
    Outros papéis podem atualizar campos não-críticos.

--- Cálculo de KPIs (NOVO v2.1) ---

  MTTR (Mean Time To Repair):
    Média das durações de WOs corretivas concluídas: (dataFimOS - dataInicioOS)
    Apenas WOs com tipo maintenance + tipoManutencao Corretiva.

  MTBF (Mean Time Between Failures):
    Por máquina: gaps de tempo entre WOs corretivas concluídas, em horas.
    Retorna null se menos de 2 falhas registadas.

  Disponibilidade:
    (720h - downtimeMes) / 720h × 100%
    Baseado em downtime declarado nos WOs do mês corrente.

  Taxa NC Frota:
    totalItensNC / totalItens × 100% calculado nos checklists recentes.

  Top 5 Falhas por Máquina:
    Contagem de WOs corretivas concluídas agrupadas por maquinaId.

--- Bridge de migração (NOVO v2.1) ---

  Leitura transparente:
    obterTodasOSBridge() executa queries em paralelo (Promise.all) em ambas
    as coleções, detecta documentos já migrados por _legacyId, mescla e
    ordena por timestamp. Evita duplicados automaticamente.

  Normalização legacy:
    normalizarLegacy() mapeia campos: statusFinal → status (statusMap),
    tipoManutencao → tipo (tipoMap), estoque.itens → materiais[],
    anexos.urlsLinks → fotos[]. Campo _source: "historico_manutencao" identifica
    a origem.

--- Sistema de Fornecedores (NOVO v2.1) ---

  Pesquisa client-side:
    buscarFornecedores(termo) faz fetch de todos os activos e filtra por
    nome, contato ou CNPJ (case-insensitive). Solução pragmática sem Algolia.

  Histórico de pedidos:
    registrarPedidoNoFornecedor() appends { poId, numero, data, valor, avaliacao }
    ao array historicoPedidos. estatisticasFornecedor() agrega: total de pedidos,
    valor total, avaliação média histórica.

  Soft delete:
    desativarFornecedor() define ativo: false. IDs de fornecedor em POs históricos
    continuam válidos. Hard delete desaconselhado.

--- Módulo Maquinário (mantido de v2.0) ---

  Gate de segurança LOTO, campos condicionais por status, formulário dual-mode
  (novo registro vs. fechar OS pendente), KPIs no histórico. Ver v2.0.

--- Lista Dinâmica de Materiais (mantido de v2.0) ---

  qty × unitPrice = totalPrice por linha (cálculo em tempo real).
  materialsCost = soma de todas as linhas. totalCost = mão de obra + materiais.
  Padrão partilhado entre os-detalhe.html e compra-detalhe.html.

--- Geração de Documentos (mantido de v2.0) ---

  SRV (db-os.js): contador sequencial por tipo service_report.
  PO (db-compras.js): contador sequencial por tipo purchase_document.
  WO novo (db-unified.js): timestamp-based SRV-YYYYMMDD-XXXX (sem condição de corrida).
  sourceData embutido no momento da geração — preview sem segunda query.

F) INTEGRATIONS
================================================================================

Serviços Firebase usados:
  Serviço              | Utilização
  ---------------------|--------------------------------------------------
  Firestore            | Base de dados principal (8 coleções)
  Firebase Storage     | Upload de fotos (maquinário, frota, limpeza por pergunta)
  Firebase Auth        | Autenticação email/password com gestão de sessão (NOVO v2.1)

Comunicação inter-módulo via Firestore:
  work_orders
    ← auto-gerado por → autoWorkOrderLimpeza() quando nota < 5.0
    ← auto-gerado por → autoWorkOrderFrota() quando NC detectado
    ← auto-gerado por → autoWorkOrderMaquina() quando máquina parada
    ← gera → documents (SRV-XXXX)
    ← dispara → purchase_orders

  purchase_orders
    ← aprovado por → aprovarPedido() (papel purchasing/admin)
    ← vinculado a → fornecedores via vincularFornecedor()
    ← regista em → fornecedores.historicoPedidos via registrarPedidoNoFornecedor()
    ← gera → documents (PO-XXXX)

  historico_manutencao + work_orders
    ← lidos juntos por → db-bridge.js (transparente para as páginas)

  users
    ← consultado por → db-auth.js em cada checkAuth() para leitura de papel

Comunicação cross-module via URL params (mantida de v2.0):
  maquinario.html      → os-detalhe.html?modo=criar&origin=machine&originId=MIS-001
  painel-frota.html    → os-detalhe.html?modo=criar&origin=fleet
  compras.html         → compra-detalhe.html?id=FIREBASE_ID
  os-detalhe.html      → compra-detalhe.html (via osOrigemId no purchase_order)


================================================================================
G) EVENT FLOW
================================================================================

------------------------------------------------------------------------
Flow 1: Acesso a Página Protegida
------------------------------------------------------------------------
  1. Browser carrega página (ex: dashboard/dashboard.html)
  2. app-dashboard.js executa import { checkAuth } from db-auth.js
  3. checkAuth("dashboard") chama getCurrentUser()
  4. getCurrentUser() dispara onAuthStateChanged uma vez (ou retorna cache)
     a. Não autenticado → redirect para auth/login.html (path relativo)
     b. Autenticado → _carregarPerfil(uid) lê users/{uid} do Firestore
  5. Papel do utilizador verificado contra ROLE_PERMISSIONS["dashboard"]
     a. Não autorizado → alert + redirect para index.html
     b. Autorizado → retorna perfil, página carrega normalmente
  6. UI adapta-se ao papel (ex: secção admin só para role === "admin")

------------------------------------------------------------------------
Flow 2: Login
------------------------------------------------------------------------
  1. Utilizador abre auth/login.html
  2. app-auth.js verifica getCurrentUser() → se já logado, redirect para index.html
  3. Submit do formulário → login(email, senha)
  4. signInWithEmailAndPassword(auth, email, senha)
  5. _carregarPerfil(user) → getDoc("users/{uid}")
  6. perfil.ativo === false → signOut() + erro "Conta desactivada"
  7. updateDoc("users/{uid}", { lastLogin: serverTimestamp() })
  8. _currentProfile = perfil (cache)
  9. Redirect para index.html

------------------------------------------------------------------------
Flow 3: Aprovação de Pedido de Compra
------------------------------------------------------------------------
  1. Utilizador (papel purchasing/admin) abre compra-detalhe.html?id=PO_ID
  2. app-compra-detalhe.js chama checkAuth("compra-detalhe")
  3. Botão "Aprovar" aparece apenas se hasRole(perfil, "purchasing", "admin")
  4. Utilizador clica "Aprovar" → modal de confirmação com campo de nota
  5. aprovarPedido(poId, { uid, nome }, nota) chamado
  6. Validação: status === "pending" (senão throw Error)
  7. Entrada adicionada ao array timeline
  8. updateDoc("purchase_orders", { status: "approved", aprovadoPor, aprovadoEm,
     aprovadoTimestamp, notaAprovacao, timeline, updatedAt })
  9. UI actualiza badge de status e timeline

------------------------------------------------------------------------

================================================================================
G-2) BUG FIXES — v2.2 (2026-04-07)
================================================================================

Todos os erros abaixo foram identificados durante revisão de código (Abril 2026)
e corrigidos na v2.2. Os módulos afetados são maquinário, frota e limpeza.

------------------------------------------------------------------------
Bug #1 — DOMContentLoaded race condition (5 arquivos)
------------------------------------------------------------------------
Causa:
  ES Modules com top-level await (await checkAuth()) suspendem a execução
  do módulo. Quando o módulo retoma, o evento DOMContentLoaded já disparou.
  Qualquer document.addEventListener("DOMContentLoaded", fn) registado após
  o await nunca executa, pois o evento já passou.

Arquivos afetados:
  app-limpeza.js            → iniciarPainelLimpeza() nunca corria
  app-frota.js              → preencherDataHoraAtual() e verificarVeiculoNaURL() nunca corriam
  app-painel-frota.js       → render completo da frota nunca corria
  app-historico-frota.js    → carregarHistorico() nunca corria
  app-historico-limpeza.js  → carregarRevisoes() nunca corria

Correção:
  Removidos todos os wrappers DOMContentLoaded. Funções chamadas diretamente
  após as declarações de variáveis (o DOM já está disponível quando o módulo
  retoma — ES modules são deferred por especificação).
  Em app-frota.js: preencherDataHoraAtual() e verificarVeiculoNaURL() movidas
  para depois das declarações de constantes DOM que elas precisam.

------------------------------------------------------------------------
Bug #2 — Missing checkAuth() em app-painel-frota.js
------------------------------------------------------------------------
Causa:
  app-painel-frota.js não tinha import de db-auth.js nem chamada checkAuth().
  Qualquer utilizador não autenticado podia ver o painel de frota.

Correção:
  Adicionado import { checkAuth } e await checkAuth("painel-frota") no topo.
  Ficheiro reescrito para remover também o DOMContentLoaded (Bug #1) e o
  inline onclick (Bug #3).

------------------------------------------------------------------------
Bug #3 — window.onclick override global (2 arquivos)
------------------------------------------------------------------------
Causa:
  app-historico-frota.js e app-historico-limpeza.js usavam window.onclick = function(event) {...}
  para fechar modais ao clicar no backdrop. Isto substituía o handler global
  de clique, podendo quebrar qualquer outro listener de clique da página.

Correção:
  Substituído por modal.addEventListener("click", (e) => { if (e.target === modal) fecharModal(); }).
  Event listener específico no elemento do modal, sem substituir o handler global.

------------------------------------------------------------------------
Bug #4 — window.abrirModalDetalhes via inline onclick em innerHTML (2 arquivos)
------------------------------------------------------------------------
Causa:
  app-historico-frota.js e app-historico-limpeza.js geravam botões com
  onclick="abrirModalDetalhes('ID')" em strings de HTML dinâmico, poluindo
  o escopo global e exigindo window.abrirModalDetalhes.

Correção:
  Botões passaram a usar data-id="${id}" sem inline handler.
  Adicionado event delegation em listaEl:
    listaEl.addEventListener("click", (e) => {
        const btn = e.target.closest(".btn-ver-detalhes");
        if (btn) abrirModalDetalhes(btn.dataset.id);
    });
  abrirModalDetalhes permanece função de módulo, não global.

------------------------------------------------------------------------
Bug #5 — window.removerPeca inline onclick em innerHTML (app-formulario.js)
------------------------------------------------------------------------
Causa:
  renderizarListaPecas() gerava <button onclick="removerPeca(${peca.id})">,
  exigindo window.removerPeca no escopo global.

Correção:
  Botão usa data-id="${peca.id}" sem handler inline.
  Event delegation em listaPecasUI substitui window.removerPeca.

------------------------------------------------------------------------
Bug #6 — window.* pollution em app-formulario-limpeza.js (5 símbolos)
------------------------------------------------------------------------
Causa:
  Os seguintes eram expostos no escopo global:
    window.fotosPorPergunta  — dicionário de File objects por pergunta
    window.respostasAuditoria — array de respostas para o submit handler
    window.calcularNotaFinal  — chamado por onchange inline em radio inputs
    window.anexarFotoLocal    — chamado por onchange inline em file inputs
    window.removerFotoLocal   — chamado por onclick inline em botões de remoção

Correção:
  - fotosPorPergunta e respostasAuditoria: variáveis de módulo (let).
  - calcularNotaFinal, anexarFotoLocal, removerFotoLocal: funções de módulo.
  - Inline handlers removidos do HTML gerado por renderizarChecklist().
  - Event delegation no container do checklist:
      container.addEventListener("change", ...) — radios e file inputs
      container.addEventListener("click", ...)  — botões de remoção de foto
  - Botão de remover foto usa data-remover-foto="${idPergunta}" em vez de onclick.
  - File input usa data-pergunta-id="${item.id}" para identificação no handler.

------------------------------------------------------------------------
Bug #7 — alert() em módulos operacionais (todos substituídos)
------------------------------------------------------------------------
Causa:
  Todas as validações e confirmações de formulário usavam alert() nativo,
  que bloqueia o event loop, tem UI inconsistente e não tem customização visual.

Arquivos afetados:
  app-frota.js, app-formulario.js, app-formulario-limpeza.js, app-historico.js

Correção:
  Cada arquivo recebeu uma função mostrarAviso(msg, tipo) local que:
  - Cria um <div> posicionado fixo no topo da página se não existir
  - Aplica cor por tipo ("erro" = vermelho, "sucesso" = verde)
  - Auto-remove após 4s com transição de opacidade
  - Não bloqueia o event loop
  Redirects que eram precedidos de alert() de erro passaram a ser
  redirects directos (sem mensagem) ou a mostrar o aviso antes de redirect.
  Sucesso após submit agora mostra aviso verde e faz redirect após 1.2s.

H) STRENGTHS AND WEAKNESSES
================================================================================

PONTOS FORTES (v2.2):

  + Autenticação real implementada: Firebase Auth com email/password,
    gestão de sessão e perfil de papel lido do Firestore.

  + RBAC completo: 4 papéis com whitelist de páginas, checkAuth() guard
    reutilizável em qualquer página, helpers hasRole() / isAdmin() etc.

  + Workflow de aprovação auditado: timeline de cada purchase_order regista
    quem fez o quê e quando, com notas opcionais.

  + Numeração de documentos sem condição de corrida: SRV-YYYYMMDD-XXXX
    em db-unified.js. (Pendente: migrar db-os.js e db-compras.js para o mesmo padrão.)

  + Bridge de migração: queries transparentes entre legacy e novo sistema.

  + Módulos operacionais estabilizados (v2.2): todos os race conditions
    DOMContentLoaded corrigidos, handlers globais removidos, formulários
    dinâmicos usam event delegation, feedback visual sem alert() nativo.
    Migração controlada em fases sem downtime.

  + Automações modulares [CF-ready]: autoWorkOrderLimpeza/Frota/Maquina
    são funções independentes que podem ser chamadas de qualquer lugar
    e migradas para Cloud Functions Triggers sem refatoração de lógica.

  + KPIs industriais reais: MTBF, MTTR, disponibilidade, compliance
    preventiva — calculados a partir dos dados reais do Firestore.

  + Regras de segurança Firestore completas: RBAC enforçado server-side,
    leitura limitada ao papel, escrita e delete controlados, catch-all nega tudo.

  + Separação clara de responsabilidades: 8 ficheiros db-*.js com domínios
    bem definidos, zero chamadas Firebase em HTML.

  + Design mobile-first mantido: containers responsivos, câmera direta
    via capture="environment", UX otimizada para touch.

  + Independência de módulos: adicionar novo módulo não toca no código existente.

PONTOS FRACOS REMANESCENTES:

  - Dados mestres estáticos não editáveis via UI: adicionar máquina, veículo
    ou zona requer editar ficheiros de código fonte. Sem painel admin CRUD.

  - Numeração sequencial em db-os.js e db-compras.js ainda vulnerável a
    condição de corrida. db-unified.js já usa timestamp; os outros precisam
    de migração.

  - Sem suporte offline: todas as operações requerem internet. Sem cache
    local, sem PWA/service worker.

  - alert() em módulos legados: app-maquinario.js e similares ainda usam
    alert() que bloqueia a thread. Módulos novos usam toast.

  - window.abrirModalDetalhes / window.removerPeca — handlers inline em HTML
    poluem o escopo global. Más práticas em ES Modules.

  - Dashboard carrega dados a cada visita: sem cache de KPIs (kpi_cache
    reservado para Fase 2 com Cloud Functions). Queries pesadas client-side.

  - Sem painel de gestão de utilizadores: criar/desativar utilizadores requer
    acesso direto ao Firestore. Falta UI admin para gestão de contas.

  - Módulo de fornecedores sem página HTML: db-fornecedores.js está completo
    mas falta fornecedores/fornecedores.html e app-fornecedores.js.

  - Sem validação de email único em utilizadores: dois perfis podem ter
    o mesmo email (Firebase Auth garante unicidade por UID, mas não há
    verificação cruzada no Firestore).

  - Limite de fotos apenas client-side: verificação JS não reforçada no
    Storage com regras de tamanho/quantidade.

I) SCALABILITY ANALYSIS
================================================================================

O sistema é modular?
  SIM — por estrutura de pastas, separação de DB e separação de Auth.
  Cada domínio operacional tem: pasta HTML, ficheiros JS, coleção Firestore,
  ficheiro DB. Novos módulos podem ser adicionados sem tocar no código existente.

O que pode crescer facilmente:
  - Adicionar novas máquinas → dados-maquinas.js (1 linha)
  - Adicionar novos veículos → dados-frota.js (1 linha)
  - Adicionar novo papel → ROLES + ROLE_PERMISSIONS em db-auth.js
  - Adicionar nova coleção → seguir padrão db-*.js existente
  - Ativar automações de WO → chamar autoWorkOrder*() após submissões existentes
  - Migrar para Cloud Functions → funções [CF-ready] já estruturadas corretamente
  - Adicionar fornecedores UI → criar HTML usando db-fornecedores.js pronto

O que requereria trabalho significativo:
  - Migrar dados mestres para Firestore: painel admin CRUD, migrar js/data/
    para coleção "catalogo" com edição via UI
  - Cache de KPIs com Cloud Functions: kpi_cache atualizado a cada 30 min;
    dashboard leria de kpi_cache em vez de calcular em tempo real
  - Numeração atómica universal: migrar db-os.js e db-compras.js para
    esquema timestamp de db-unified.js, ou usar transações Firestore
  - Capacidade offline: PWA (service worker, IndexedDB, sync em background)
  - Multi-tenancy: caminhos por empresa ou projetos Firebase separados
  - Painel admin de utilizadores: UI para criar/desativar/mudar papel
  - Integração com ERP: API layer entre dados-maquinas.js e sistema externo

O que precisaria de refatoração:
  Área                     | Problema                          | Refatoração
  -------------------------|-----------------------------------|----------------------------------------
  Handlers inline          | window.removerPeca etc.           | Substituir onclick por delegação de eventos
  Numeração docs           | Condição de corrida em os/compras | Migrar para schema timestamp db-unified
  alert() em módulos legad | Bloqueia UI                       | Sistema de toast unificado
  db.js monolítico         | Mistura 3 módulos                 | Separar em db-maquinas, db-frota, db-limpeza
  formatarData() duplicado | Em os.js, compras.js, unified.js  | Extrair para js/core/utils.js


================================================================================
J) FINAL SUMMARY
================================================================================

SIGA é uma plataforma de operações industriais mobile-first, inteiramente
client-side, construída sobre Firebase Firestore, Storage e Auth, usando
ES Modules vanilla sem build system.

Arquitetura em 3 camadas: HTML (UI), app-*.js (page controllers), db-*.js
(Firebase abstraction). 8 ficheiros de camada DB cobrem domínios distintos:
auth, OS unificado, OS legado, compras, fornecedores, dashboard, bridge de
migração e módulos operacionais (maquinário/frota/limpeza).

Módulos operacionais: Maquinário, Frota, Limpeza, Ordens de Serviço, Compras,
Documentos — cada um com HTML, JS, CSS e coleção Firestore próprios.
Módulos de sistema: Auth/RBAC, Fornecedores, Dashboard de KPIs, Unified WO.

Fluxo de dados: catálogo estático em js/data/ (sem rede), dados operacionais
em Firestore com timestamp duplo (serverTimestamp + Date.now()), fotos no Storage.
Auth real via Firebase com papéis (admin, maintenance, operations, purchasing)
lidos do Firestore, guardando cada página com checkAuth().

Funcionalidade mais sofisticada: integração OS + Compras + Documentos com
workflow de aprovação auditado, fornecedores vinculados, automações [CF-ready]
de criação de WO a partir de eventos de outros módulos, KPIs industriais
(MTBF, MTTR, disponibilidade) calculados client-side a partir de dados reais.

Bridge de migração: leitura transparente de historico_manutencao e work_orders
simultaneamente, com utilitário de migração em fases sem downtime.

Regras de segurança: firestore.rules com RBAC por coleção enforçado server-side,
catch-all nega tudo o que não está explicitamente permitido.

Limitações para produção:
  - Dados mestres não editáveis via UI
  - Numeração sequencial em módulos legados (risco de corrida)
  - Sem suporte offline
  - Sem painel admin de utilizadores
  - Dashboard calcula KPIs em tempo real (sem cache)
  - alert() em módulos legados (UX)

Roadmap imediato:
  Fase 2: Cloud Functions (automações como triggers, cache de KPIs agendado)
  Fase 2: Migração de dados legacy via migrarHistoricoParaWorkOrders()
  Fase 3: Painel admin de utilizadores, fornecedores.html, migração de dados mestres
